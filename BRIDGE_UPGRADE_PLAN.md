# Buddy ↔ LLM 桥梁强化方案 & 开发计划

> 核心理念：LLM 是老师，Buddy 是学生。桥的质量决定了学生能学到多少。
> 最终目标：学生学会做事，不再每件事都问老师。

---

## 一、架构总览

### 1.1 师生关系模型

```
┌───────────────────────┐              ┌───────────────────────┐
│        Buddy          │              │        LLM            │
│       (学生)           │              │      (老师)            │
│                       │              │                       │
│  ┌─────────────────┐  │   提问/上下文  │  ┌─────────────────┐  │
│  │  Prompt 预算管理  │──┼─────────────→│  │   理解 + 生成    │  │
│  └─────────────────┘  │              │  └─────────────────┘  │
│                       │  文本+工具调用  │                       │
│  ┌─────────────────┐  │←─────────────┼  ┌─────────────────┐  │
│  │  响应格式标准化   │  │              │  │   任意格式输出    │  │
│  └─────────────────┘  │              │  └─────────────────┘  │
│                       │              │                       │
│  ┌─────────────────┐  │              │  不记得学生             │
│  │  通用工具调用引擎  │  │              │  有知识但无记忆         │
│  └─────────────────┘  │              │  每次重新认识           │
│                       │              │                       │
│  记忆/经验/性格/情绪   │              └───────────────────────┘
│  越用越聪明           │
│  最终可以自己做事      │
└───────────────────────┘
```

### 1.2 四段桥梁

| 桥段 | 职责 | 组件 |
|------|------|------|
| 学生→老师（提问） | 精炼、有重点地表达 | PromptBudgetManager |
| 老师→学生（回答） | 理解各种方言 | ResponseNormalizer |
| 学生执行（动手） | 解析工具调用 + 安全执行 | UniversalToolCaller |
| 学生消化（成长） | 记录、学习、内化 | Experience Compiler + Router |

## 二、组件设计

### 2.1 Provider 注册表（provider-registry.ts）

**解决的问题**：只认 OpenAI 兼容格式，听不懂其他老师说话。

```
┌─────────────┬──────────┬──────────┬──────────┬───────────┬───────────┐
│ Provider    │ 工具调用  │ 流式输出  │ 结构化   │ 最大上下文  │ toolChoice│
├─────────────┼──────────┼──────────┼──────────┼───────────┼───────────┤
│ OpenAI      │ 原生 ✅  │ ✅       │ ✅       │ 128K      │ strict    │
│ Claude      │ 原生 ✅  │ ✅       │ ✅       │ 200K      │ auto      │
│ Gemini      │ 原生 ✅  │ ✅       │ ✅       │ 1M        │ auto      │
│ DeepSeek    │ 原生 ✅  │ ✅       │ ⚠️ 降级  │ 64K       │ auto      │
│ Ollama      │ 部分 ⚠️  │ ✅       │ ⚠️ 降级  │ 32K       │ false     │
│ Custom      │ 自动探测 │ ✅       │ ⚠️ 降级  │ 32K       │ auto      │
└─────────────┴──────────┴──────────┴──────────┴───────────┴───────────┘
```

能力降级策略：
- 不支持 tool calling → Prompt 模拟（JSON 输出 + 解析）
- 不支持 structured output → Prompt 要求 JSON + 手动校验
- 不支持 toolChoice → 不传该参数
- 不支持并行调用 → 每步只允许一个工具

### 2.2 通用工具调用引擎（universal-tool-caller.ts）

**解决的问题**：不同模型输出工具调用的格式千差万别。

**9 种解析策略**（按优先级）：

| 优先级 | 策略 | 适配模型 |
|--------|------|---------|
| 100 | ```json ``` 代码块 | 大多数模型 |
| 95 | <tool_call> 标签 | Qwen 系列 |
| 90 | <tool_call>{...} 单行 | 部分国产模型 |
| 85 |  参数化标签 | 特定模型 |
| 84 | 多参数 invoke | 特定模型 |
| 80 | [TOOL_CALL] 方括号 | 特定系统 |
| 70 | 纯 JSON（整段） | 通用 |
| 60 | JSON 嵌在文本中 | 通用兜底 |
| 10 | 自然语言意图推断 | 终极兜底 |

**参数修复**：
- 类型强制转换（字符串→数字、单值→数组）
- 默认值填充
- 多余字段剥离
- Levenshtein 模糊匹配工具名

**定位**：纯执行器。只管"解析格式 + 执行工具 + 返回结果"。不参与决策和学习。

### 2.3 响应格式标准化器（response-normalizer.ts）

**解决的问题**：不同 API 的响应格式差异。

三层处理：
- Layer 1：Ollama vs OpenAI 的 HTTP 响应包装差异
- Layer 2：tool_calls 结构差异（缺 id、缺 type、arguments 类型不一致）
- Layer 3：文本中混杂的工具调用（模型把工具调用写在自然语言里）

### 2.4 Prompt 预算管理器（prompt-budget.ts）

**解决的问题**：System Prompt 拼接无上限，每次重新自我介绍。

优先级分配：

| 段 | 优先级 | 必须保留 | 说明 |
|----|--------|----------|------|
| 核心指令 + 安全规则 | 100 | ✅ | 不可省略 |
| 信任度/权限边界 | 95 | ✅ | 安全相关 |
| 当前工具列表 | 80 | ❌ | 超预算截断 |
| 人格属性 | 70 | ❌ | 影响风格 |
| 情绪状态 | 60 | ❌ | 可丢弃 |
| 用户认知画像 | 50 | ❌ | 可丢弃 |
| 记忆检索结果 | 40 | ❌ | 可丢弃 |
| 领域知识 | 30 | ❌ | 可丢弃 |
| 技能列表 | 20 | ❌ | 可丢弃 |

## 三、探索 vs 利用：学生的学习策略

### 3.1 三种沟通模式

```
探索模式（每次都问老师）
  → 学生刚入学 / 遇到新任务
  → Token 消耗高，但能学到新东西

利用模式（学会了自己做）
  → 学生熟练后
  → Token 消耗为零，但遇到新情况不会处理

混合模式（不确定时问，确定时自己做）
  → 最优策略
  → 平衡学习效率和执行效率
```

### 3.2 自适应置信度路由

基于学术研究（AgentRR、APC）的三层路由：

```
输入 → 新颖度检测（和历史经验的相似度分布）
  │
  ├─ 极低新颖度（和某个经验高度相似）
  │  → exp_direct：零 LLM，直接执行
  │
  ├─ 中等新颖度（和多个经验部分相似）
  │  → exp_verified：执行 + LLM 验证
  │
  ├─ 高新颖度（和任何经验都不太像）
  │  → llm_with_hint：LLM 为主，经验参考
  │
  └─ 极高新颖度（完全没见过）
     → llm_only：纯 LLM + 强制学习
```

### 3.3 学习触发条件

```
每次走 LLM 的交互 → 尝试编译经验
  → 编译成功 → 写入经验图谱（初始置信度低）
  → 编译失败 → 跳过，不硬学

经验执行后 → 置信度更新
  → 成功 → 置信度 +Δ
  → 失败 → 置信度 -Δ
  → 连续失败 → 淘汰

每 N 次高置信度执行 → 抽查一次走 LLM
  → 验证经验是否还有效
  → 防止"学会了坏习惯"
```

### 3.4 经验生命周期

```
新生期（置信度 < 0.3）
  → 只作为 hint，不独立执行
  → 需要 LLM 验证

成长期（置信度 0.3-0.5）
  → 可以执行，但要 LLM 事后验证
  → 成功率记录

成熟期（置信度 0.5-0.8）
  → 可以独立执行
  → 偶尔抽查

完全体（置信度 > 0.8 + 成功 ≥ 3 次）
  → 直接执行，零 LLM
  → 定期抽查

淘汰（连续失败 ≥ 3 次）
  → 置信度归零
  → 降级回新生期或删除
```

## 四、学术研究支撑

### 4.1 AgentRR（上海交大，arxiv:2505.17716）

**核心贡献**：Record & Replay 范式引入 Agent 系统

- 多层次经验抽象（具体动作 → 工作流 → 抽象策略）
- Check Function（回放时的安全检查）
- 经验仓库（可跨 agent 共享）

**Buddy 可借鉴**：增加经验的抽象层级，Check Function 扩展到工作流级别。

### 4.2 Agentic Plan Caching（斯坦福，NeurIPS 2025，arxiv:2506.14852）

**核心贡献**：计划模板缓存

- 缓存"计划模板"而非"输入输出对"
- 关键词匹配 + 轻量模型适配参数
- 成本降低 50.31%，延迟降低 27.28%

**Buddy 可借鉴**：经验单元的参数化适配，量化 token 节省。

### 4.3 A-MEM（NeurIPS 2025，arxiv:2502.12110）

**核心贡献**：自组织记忆（Zettelkasten 方法）

- 动态索引和链接
- 记忆进化（新记忆更新旧记忆的理解）
- Agent 驱动的组织方式

**Buddy 可借鉴**：STMP 记忆系统的动态链接和进化机制。

### 4.4 H-MEM（arxiv:2507.22925）

**核心贡献**：层级记忆 + 索引路由

- 按语义抽象程度分层
- 位置索引编码
- 逐层路由检索（不用全局相似度计算）

**Buddy 可借鉴**：给 STMP 记忆加位置索引编码，提升检索效率。

### 4.5 GRPO 工具调用训练（arxiv:2509.04518）

**核心贡献**：用强化学习训练小模型的工具调用能力

- 奖励函数：JSON 格式正确性 + 工具选择准确性 + 参数精确性

**Buddy 可借鉴**：未来训练自有小模型时参考。

### 4.6 MAE 多 Agent 进化（arxiv:2510.23595）

**核心贡献**：自我博弈进化

- Proposer/Solver/Judge 三角互相博弈
- 不需要外部监督数据

**Buddy 可借鉴**：经验模型的自我验证和进化机制。

## 五、开发计划

### Phase 1：桥基 — 让学生听得懂（1-2天）

**目标**：多 Provider 兼容 + LLM 参数控制

| # | 任务 | 文件 | 改动 |
|---|------|------|------|
| 1.1 | LLM 参数透传 | types.ts, llm.ts | 新增 temperature/maxOutputTokens 等 6 个参数 |
| 1.2 | Provider 注册表 | 新建 provider-registry.ts | 6 种 Provider + 能力定义 + 动态降级 |
| 1.3 | 响应格式标准化 | 新建 response-normalizer.ts | 6 种格式解析策略 |
| 1.4 | 安装依赖 | package.json | 新增 @ai-sdk/anthropic, @ai-sdk/google |

**验收标准**：OpenAI / Claude / Gemini / DeepSeek / Ollama 均可连接并对话。

### Phase 2：桥面 — 让学生问得好（2-3天）

**目标**：Prompt 预算管理 + 动态工具裁剪

| # | 任务 | 文件 | 改动 |
|---|------|------|------|
| 2.1 | Prompt 预算管理器 | 新建 prompt-budget.ts | Token 预算 + 优先级 + 截断 |
| 2.2 | 改造 buildContext | message-processor.ts | 用预算管理器组装 Prompt |
| 2.3 | activeTools 动态裁剪 | message-processor.ts | 按意图筛选工具子集 |
| 2.4 | 意图分类 | 新增 intent-classifier.ts | Output.choice 做快速意图判断 |

**验收标准**：System Prompt 不超过预算上限，低优先级段自动丢弃。

### Phase 3：桥栏 — 让学生记得住（2-3天）

**目标**：通用工具调用 + 结构化输出 + 即时学习

| # | 任务 | 文件 | 改动 |
|---|------|------|------|
| 3.1 | 通用工具调用引擎集成 | llm.ts | 原生/模拟双路径 + 自动降级 |
| 3.2 | 工具调用修复 | llm.ts | experimental_repairToolCall |
| 3.3 | 结构化输出 | extractor.ts | Output.object + Zod Schema |
| 3.4 | 即时知识标记 | message-processor.ts | 对话后立即标记知识原子 |
| 3.5 | 经验路由增强 | experience-router.ts | 增加新颖度检测 |

**验收标准**：不支持 tool calling 的模型也能正常使用工具；知识提取实时触发。

### Phase 4：桥灯 — 让学生形成判断（3-5天）

**目标**：多层次经验 + 自适应学习 + 经验进化

| # | 任务 | 文件 | 改动 |
|---|------|------|------|
| 4.1 | 多层次经验抽象 | experience-compiler.ts | 具体→工作流→策略三层 |
| 4.2 | Check Function | 新增 check-function.ts | 工作流完整性检查 |
| 4.3 | 自适应置信度路由 | experience-router.ts | Thompson Sampling 策略 |
| 4.4 | 经验进化器增强 | experience-evolver.ts | 置信度更新 + 淘汰 + 抽查 |
| 4.5 | 计划模板参数化 | experience-executor.ts | 轻量模型适配参数 |
| 4.6 | 经验效果量化 | 新增 metrics.ts | 记录 token 节省、成功率 |

**验收标准**：用得越久 LLM 调用越少；有量化数据证明成本降低。
